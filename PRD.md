# Travel Diary — 产品需求文档（PRD）

## 1. 执行摘要

我们要做一个 **Web 端旅行规划工具（Travel Diary）**，核心是一份「旅行计划」贯穿旅行全程：出发前用它规划路线和行程安排，旅途中在计划上补充照片、文字和花费记录，行程结束后自动生成账单并支持导出。产品最大的差异点是**内置地图**，能根据交通方式（公共交通/自驾）显示真实路线，让用户直观判断路线是否合理。解决现有工具信息分散、需要在多个软件间反复跳转、缺乏地图支持的核心痛点。

---

## 2. 问题陈述

### 谁有这个问题？

喜欢自由行的旅行者，尤其是会花大量时间做旅行规划的人。不限人数场景（个人、情侣、家庭、朋友结伴均适用）。

### 问题是什么？

旅行规划时，用户需要在**多个工具之间反复跳转**：

1. 去社交媒体（小红书、YouTube 等）搜集景点信息
2. 打开地图 App 查看景点位置和路线
3. 用 Excel 做日程表格
4. 用无边际 / 备忘录整理零散资料
5. 手动截图地图，再标点画线来规划路线

这些信息**分散在不同工具中**，没有一个地方能统一收纳。更关键的是，**现有的旅行规划工具大多不带地图功能**，用户无法直观判断路线是否存在往返或浪费。

### 为什么痛苦？

- **效率低**：在 4-5 个软件之间反复切换，信息容易遗漏
- **路线不直观**：没有地图支持，只能靠截图手动标点，无法判断路线合理性
- **信息碎片化**：收集的大量资料散落各处，难以整理和复用
- **旅途中不便**：出行时需要翻找多个 App 才能看到完整行程

### 证据

- 用户（产品发起人）自身是重度旅行规划者，每次规划都经历上述痛苦流程
- 市面主流旅行工具（小红书收藏、旅行清单类 App）普遍缺乏地图路线可视化功能

---

## 3. 目标用户

### 主要用户画像：自由行规划者

- **典型场景**：出国自由行（如日本旅行），需要详细规划每日行程
- **行为特征**：
  - 会花大量时间在社交媒体上搜集景点信息
  - 习惯用表格或清单工具整理行程
  - 重视路线效率，不想走冤枉路
  - 旅途中会记录花费
- **技术水平**：普通用户，不需要复杂功能
- **核心诉求**：一个地方搞定规划、地图、记录、记账

### 用户规划旅行的典型流程

1. **确定方向与核心景点** — 从社交媒体收集必去的点
2. **在地图上查看路线** — 按效率串联景点
3. **设置每日主题** — 如「森林徒步」「城市购物」
4. **细化行程安排** — 每天的景点、时间、交通方式
5. **准备清单** — 签证、物品等出行准备

---

## 4. 战略背景

### 为什么做这个？

- 市面缺乏一个**同时具备行程规划 + 地图路线可视化 + 旅途记录 + 记账**的轻量工具
- 现有工具要么功能太重（专业旅行平台），要么太轻（纯清单/备忘录），都缺乏地图路线支持
- 产品发起人自身有强烈需求，近期日本旅行可直接验证产品

### 为什么是现在？

- 用户近期有日本旅行计划，产品可以在真实场景中立即使用和验证
- Web 技术成熟，可以快速实现 MVP

## 5. 解决方案概述

### 核心理念：计划即日记

整个产品围绕一份「旅行计划」展开，这份计划贯穿 **行前 → 行中 → 行后** 三个阶段：

| 阶段 | 用途 |
|------|------|
| 行前 | 规划路线、安排每日行程、准备清单 |
| 行中 | 在计划上补充照片、文字、记录花费 |
| 行后 | 查看完整旅行记录、账单汇总、导出花费表格 |

### 产品结构

```
首页（旅行计划列表）
├── 计划 A「日本关西 7 日游」
├── 计划 B「云南 5 日游」
└── + 新建计划

进入某个计划后，包含三个页面：
├── 📋 日程计划页（核心页面，所有编辑在此完成）
├── 🗺️ 地图页（只读可视化，计划内容自动映射）
└── 💰 旅行账单页（自动汇总，支持导出）
```

### 页面详细说明

#### 5.1 日程计划页（核心页面）

这是用户花最多时间的页面，所有信息的录入都在这里完成。

```
日程计划页
├── 准备清单（顶部，可收起的下拉列表）
│   ├── ☐ 护照
│   ├── ☐ 签证
│   ├── ☐ WiFi / 电话卡
│   └── ☐ ...
│
├── Day 1 - 3/15 周六「大阪城市探索」（每日主题）
│   ├── 09:00 大阪城公园 📍
│   │   ├── 备注（文字）
│   │   ├── 照片
│   │   └── 花费：¥500 - 门票
│   │
│   │   🚇 地铁 → 30min
│   │
│   ├── 12:00 道顿堀午餐 📍
│   │   ├── 备注
│   │   ├── 照片
│   │   └── 花费：¥2000 - 餐饮
│   │
│   │   🚶 步行 → 10min
│   │
│   └── 15:00 心斋桥购物 📍
│       ├── 备注
│       ├── 照片
│       └── 花费：¥5000 - 购物
│
├── Day 2 - 3/16 周日「奈良森林徒步」
│   └── ...
│
└── Day 3 ...
```

**关键交互：**
- 每个景点可以展开，填写备注、上传照片、记录花费
- 景点之间的交通方式由用户手动选择（地铁/巴士/驾车/步行等），系统根据交通方式自动生成两点之间的路线
- 准备清单在页面顶部，默认收起，点击展开

#### 5.2 地图页（只读可视化）

日程计划中的所有地点和路线自动映射到地图上，**不在地图页做任何编辑操作**。

**功能：**
- 显示所有目的地标记点
- 根据交通方式显示点与点之间的真实路线：
  - 公共交通（地铁/巴士）→ 显示公共交通路线
  - 自驾 → 显示公路驾车路线
  - 步行 → 显示步行路线
- 点击某个地点，弹出该点的详细信息（所属哪天、时间、备注等）
- 可按天筛选显示（只看 Day 1 的路线，或查看全部）

#### 5.3 旅行账单页（自动汇总）

**纯展示页面**，所有花费数据来源于日程计划页中每个景点填写的花费，不可在账单页手动编辑。

```
旅行账单
├── 总花费：¥38,000
│
├── 分类汇总
│   ├── 交通：¥12,000
│   ├── 餐饮：¥10,000
│   ├── 门票：¥5,000
│   ├── 住宿：¥8,000
│   └── 购物：¥3,000
│
├── 按天明细
│   ├── Day 1 大阪城市探索：¥7,500
│   │   ├── 大阪城公园 - 门票 ¥500
│   │   ├── 道顿堀午餐 - 餐饮 ¥2,000
│   │   └── 心斋桥购物 - 购物 ¥5,000
│   ├── Day 2 奈良森林徒步：¥4,200
│   └── ...
│
└── [导出表格]
```

**行为说明：**
- 行前（无花费记录）：显示 ¥0，提示「还没有花费记录」
- 行中：随着用户在景点中记录花费，实时更新汇总
- 行后：完整账单，支持导出为表格文件

## 6. 成功指标

由于这是个人项目，不设置传统的商业指标。以**自用验证**为主：

### 主要指标
- **能否完整规划一次日本旅行**：用这个工具完成全部行程规划，不需要借助其他工具（地图截图、Excel 等）
- **旅途中实际使用**：行程中能方便地查看计划、记录花费和照片

### 次要指标
- 地图路线与实际路线的吻合度
- 页面加载和交互的流畅度

## 7. 用户故事与需求

### Epic

作为一个旅行规划者，我希望有一个集成地图的旅行规划工具，让我在一个地方完成行程规划、旅途记录和花费管理。

### 用户故事

#### 旅行计划管理

**US-01：创建旅行计划**
作为用户，我想创建一个新的旅行计划，以便开始规划我的旅程。

验收标准：
- [ ] 可以在首页点击「新建计划」
- [ ] 输入计划名称（如「日本关西 7 日游」）
- [ ] 设置旅行起止日期
- [ ] 创建后进入日程计划页

**US-02：管理旅行计划列表**
作为用户，我想在首页查看和管理我所有的旅行计划。

验收标准：
- [ ] 首页展示所有旅行计划卡片
- [ ] 可以点击进入某个计划
- [ ] 可以删除计划

#### 日程计划（核心功能）

**US-03：管理每日行程**
作为用户，我想为旅行的每一天设置主题并安排景点。

验收标准：
- [ ] 根据起止日期自动生成每日分组（Day 1, Day 2...）
- [ ] 每天可以设置一个主题名称（如「城市探索」「森林徒步」）
- [ ] 可以在每天下添加景点/目的地
- [ ] 景点需填写：名称、时间、地址（用于地图定位）
- [ ] 景点可以排序、删除

**US-04：设置景点间交通方式**
作为用户，我想为相邻景点之间选择交通方式，以便在地图上看到对应路线。

验收标准：
- [ ] 相邻景点之间可以选择交通方式：地铁/巴士/驾车/步行/其他
- [ ] 选择后系统自动生成对应的路线（用于地图展示）
- [ ] 如果路线不准确，用户可以修改交通方式重新生成

**US-05：记录旅途内容**
作为用户，我想在每个景点下记录文字、照片和花费。

验收标准：
- [ ] 每个景点可展开，显示详情编辑区
- [ ] 可以添加文字备注
- [ ] 可以上传照片（存储在本地）
- [ ] 可以记录花费（金额 + 分类）
- [ ] 花费分类包括：交通、餐饮、门票、住宿、购物、其他

**US-06：管理准备清单**
作为用户，我想在日程页面顶部维护一个出行准备清单。

验收标准：
- [ ] 页面顶部有可收起/展开的准备清单区域
- [ ] 可以添加、删除、勾选清单项
- [ ] 默认收起状态

#### 地图功能

**US-07：查看地图可视化**
作为用户，我想在地图页面看到所有景点和路线，以便直观判断路线是否合理。

验收标准：
- [ ] 所有景点以标记点形式显示在地图上
- [ ] 相邻景点之间根据选择的交通方式显示对应路线
  - 公共交通 → 公共交通路线
  - 驾车 → 公路路线
  - 步行 → 步行路线
- [ ] 点击景点标记，弹出该景点的详细信息
- [ ] 可以按天筛选显示

#### 旅行账单

**US-08：查看旅行账单**
作为用户，我想查看旅行的花费汇总，了解钱花在了哪里。

验收标准：
- [ ] 自动汇总所有景点中记录的花费
- [ ] 显示总花费金额
- [ ] 按分类（交通、餐饮等）显示汇总
- [ ] 按天显示花费明细
- [ ] 无花费时显示「还没有花费记录」

**US-09：导出账单**
作为用户，我想将旅行账单导出为表格文件，方便保存和分享。

验收标准：
- [ ] 提供「导出表格」按钮
- [ ] 导出为 Excel 或 CSV 格式
- [ ] 包含按天、按分类的花费明细

## 8. 范围外（本期不做）

| 功能 | 原因 |
|------|------|
| 用户注册 / 登录 / 云同步 | 先本地存储验证功能，后续再加 |
| 地图上直接编辑（添加/删除景点）| 保持地图页简洁，所有编辑在日程计划页完成 |
| 导航功能 | 只做路线可视化，不做实时导航 |
| 多币种支持 | 极简记账，先用单一币种 |
| 多人协作 / 分摊记账 | 后续考虑 |
| 酒店 / 机票预订集成 | 不做预订功能 |
| AI 智能推荐路线 | 后续考虑 |
| 社交分享功能 | 后续考虑 |
| 移动端 App / 小程序 | Web 先行，验证后再扩展 |

## 9. 技术方案

| 项目 | 选型 | 说明 |
|------|------|------|
| 前端框架 | Vue 3 + Vite | 学习友好，中文生态丰富 |
| UI 组件库 | Element Plus | 提供现成的 UI 组件 |
| 地图服务 | Google Maps JavaScript API | 海外旅行场景（日本等）数据精准，公交路线覆盖好 |
| 数据存储 | localStorage | 先本地存储，后续再接后端 |
| 后续扩展 | uni-app | 可扩展至小程序和移动 App |

### Google Maps API 需要的能力

| API | 用途 |
|-----|------|
| Maps JavaScript API | 地图显示 |
| Directions API | 根据交通方式生成路线 |
| Places API | 地点搜索和定位 |
| Geocoding API | 地址转坐标 |
